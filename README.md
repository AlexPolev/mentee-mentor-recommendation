Проект для тестирования рекомендательной системы подбора менторов для менти в условиях отсутствия реальных пользовательских данных.

Система работает с симулированными JSON-профилями менти и менторов, отправляет выбранный профиль менти и список кандидатов в YandexGPT 5 Lite, а затем получает список рекомендованных менторов с объяснением логики выбора.

## Цель проекта

Цель проекта — проработать техническую часть рекомендации для сервиса менторства:

1. определить характеристики профилей ментора и менти на основе анализа конкурентов;
2. сгенерировать тестовые профили с разной степенью заполненности;
3. реализовать первый подход к рекомендательной системе через YandexGPT;
4. получить объяснимые рекомендации;
5. протестировать, как качество рекомендаций меняется при полной, средней и низкой заполненности профиля.

## Реализованный подход

### Подход 1. LLM-ranking без заранее заданной формулы

В текущей версии YandexGPT получает:

- один выбранный профиль менти;
- ограниченный список профилей менторов;
- инструкцию самостоятельно определить критерии рекомендации;
- требование вернуть top-N менторов и объяснить выбор.

Модель сама анализирует профили и возвращает:

- общую логику рекомендации;
- список рекомендованных менторов;
- оценку релевантности;
- объяснение по каждому ментору;
- совпавшие факторы;
- возможные риски и ограничения.

Важно: в текущей версии нет собственной формулы скоринга, hard filters и предварительного отбора кандидатов. Это baseline-подход для первичного тестирования.

## Данные

В проекте используются два основных JSON-файла:

```text
data/mentees.json
data/mentors.json
```

Файлы содержат симулированные профили с разной заполненностью:

```text
profile_completeness = high
profile_completeness = medium
profile_completeness = low
```

Это нужно для тестирования того, как YandexGPT меняет рекомендации и объяснения в зависимости от полноты данных.

### Примеры контрольных профилей

```text
mentee_control_pm_transition
```

Контрольный кейс: аналитик хочет перейти в Product Manager.

```text
mentor_control_pm_transition
```

Ожидаемо релевантный ментор для этого кейса.

```text
mentee_control_low_info
```

Контрольный кейс с низкой заполненностью профиля: почти только свободный текст про неудачные собеседования.

```text
mentor_control_career_interview
```

Потенциально релевантный ментор для кейса с собеседованиями и резюме.

## Тестирование рекомендаций

Тестирование проводится вручную на группах профилей с разной заполненностью.

### Группы тестов

```text
high   — полностью заполненные профили
medium — частично заполненные профили
low    — сильно неполные профили
```

Рекомендуемый минимум:

```text
5 high-профилей
5 medium-профилей
5 low-профилей
```

Оптимально:

```text
10 high-профилей
10 medium-профилей
10 low-профилей
```

### Что фиксировать при тестировании

Для каждого прогона нужно записывать:

1. ID выбранного менти;
2. заполненность профиля: high / medium / low;
3. какие менторы попали в top-N;
4. релевантен ли top-1;
5. есть ли нерелевантные менторы в top-5;
6. насколько логично объяснение YandexGPT;
7. учитывает ли модель неполные данные;
8. указывает ли модель риски;
9. общий комментарий.

### Рекомендуемая шкала оценки

#### Релевантность top-1

```text
0 — нерелевантен
1 — слабо релевантен
2 — частично релевантен
3 — в целом релевантен
4 — хорошо релевантен
5 — идеально релевантен
```

#### Качество top-5

```text
0 — большинство рекомендаций нерелевантны
1 — релевантен только top-1
2 — релевантны 1–2 ментора
3 — релевантны 2–3 ментора
4 — релевантны 4 ментора
5 — все 5 рекомендаций релевантны
```

#### Объяснимость

```text
0 — объяснение отсутствует
1 — объяснение общее и не связано с данными
2 — объяснение частично связано с профилем
3 — объяснение использует 1–2 реальных признака
4 — объяснение использует несколько признаков
5 — объяснение точное, конкретное и проверяемое по JSON
```

#### Работа с неполными данными

```text
0 — модель игнорирует неполноту
1 — делает уверенные выводы без данных
2 — иногда указывает нехватку данных
3 — корректно указывает часть рисков
4 — хорошо отмечает неизвестные поля
5 — осторожно рекомендует и явно пишет, что нужно уточнить
```

## Ограничения текущего подхода

Текущая реализация является первым baseline-вариантом.

Ограничения:

1. YandexGPT сам определяет критерии рекомендации, поэтому результат может быть нестабильным.
2. Нет собственной формулы скоринга.
3. Нет hard filters по языку, бюджету, доступности и уровню.
4. Нет предварительного отбора кандидатов из всего списка менторов.
5. В текущей версии в запрос к YandexGPT передается ограниченный список менторов, а не весь файл.
6. Объяснения модели нужно проверять: они могут звучать убедительно, даже если совпадение слабое.

## Возможные следующие шаги

1. Добавить предварительный отбор кандидатов перед YandexGPT.
2. Реализовать hard filters:
   - язык;
   - бюджет;
   - доступность;
   - уровень менти;
   - формат работы.
3. Реализовать weighted scoring.
4. Сравнить подходы:
   - naive LLM-ranking;
   - candidate selection + LLM-ranking;
   - hard filters + weighted score;
   - hybrid scoring + LLM explanation.
5. Сохранять результаты тестов в CSV.
6. Добавить страницу истории тестовых прогонов.
7. Добавить автоматический расчет метрик качества рекомендаций.

## Связь с анализом конкурентов

В конкурентном анализе были выделены ключевые признаки, которые чаще всего используются сервисами менторства:

- цель менти;
- текстовый запрос или проблема;
- навыки;
- опыт;
- уровень;
- формат взаимодействия;
- доступность;
- часовой пояс;
- язык;
- бюджет;
- отзывы и надежность;
- коммуникационные предпочтения.

Текущая реализация использует эти признаки как входные данные для YandexGPT. Следующий этап — формализовать их в собственной гибридной модели рекомендаций.
